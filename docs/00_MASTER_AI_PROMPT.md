# CHAPTER 7 — ARCHITECTURE RULES

---

## 7.1 Purpose

The NIAHAIR ERP architecture defines how every part of the system is organized and interacts.

Architecture ensures consistency, scalability, maintainability, and long-term stability.

Every implementation must preserve the official architecture.

---

## 7.2 Architecture Is Mandatory

Never violate the project's architecture.

Do not introduce shortcuts that bypass the defined layers.

Architecture always takes precedence over implementation convenience.

---

## 7.3 Official Architecture

Every module follows the same architecture.

```text
Client

↓

Routes

↓

Middleware

↓

Controller

↓

Service

↓

Repository

↓

Prisma ORM

↓

Database
```

No layer may bypass another.

---

## 7.4 Layer Responsibilities

Each layer has exactly one responsibility.

| Layer | Responsibility |
|--------|----------------|
| Routes | Register endpoints and middleware |
| Middleware | Authentication, authorization, validation |
| Controller | HTTP request and response |
| Service | Business Logic |
| Repository | Database access |
| Prisma | ORM |
| Database | Data persistence |

Never mix responsibilities.

---

## 7.5 Forbidden Layer Violations

The following are prohibited.

❌ Controller accessing Prisma directly.

❌ Controller containing Business Logic.

❌ Repository calling another Repository from a different module.

❌ Repository calling Controller.

❌ Service generating HTTP Response.

❌ DTO containing Business Logic.

❌ Validation containing Business Rules.

Every layer must remain isolated.

---

## 7.6 Dependency Direction

Dependencies always flow downward.

```text
Controller

↓

Service

↓

Repository

↓

Database
```

Never reverse this dependency.

---

## 7.7 Cross-Module Communication

Modules communicate only through Services.

Correct.

# CHAPTER 10 — DATABASE RULES

---

## 10.1 Purpose

This chapter defines the mandatory rules for designing, modifying, and interacting with the database within the NIAHAIR ERP project.

The database is the single source of truth for all persistent business data.

Every database change must preserve data integrity, consistency, and long-term maintainability.

---

## 10.2 Database Philosophy

The database is not just a storage engine.

It represents the business domain.

Every table, relation, and constraint should exist because of a business requirement.

Never design the database around implementation convenience.

---

## 10.3 Single Source of Truth

Business data must exist in only one authoritative location.

Avoid duplicate storage.

Avoid duplicated business information.

If the same data appears in multiple places, clearly define which one is authoritative.

---

## 10.4 ORM Standard

All database access must use Prisma ORM.

Never bypass Prisma unless explicitly approved.

Raw SQL should only be used when:

- performance requires it
- Prisma cannot support the operation
- architectural approval has been given

---

## 10.5 Migration Rules

Every schema change requires a migration.

Never modify the production schema manually.

Always:

- generate migration
- review migration
- test migration
- document migration

Migration history must remain complete.

---

## 10.6 Data Integrity

Always protect:

- Primary Keys
- Foreign Keys
- Unique Constraints
- Check Constraints
- Transactions

Never sacrifice integrity for convenience.

---

## 10.7 Relationships

Relationships must reflect real business relationships.

Always define:

- One-to-One
- One-to-Many
- Many-to-Many

based on Business Rules.

Never invent relationships.

---

## 10.8 Soft Delete

Use Soft Delete where required by Business Rules.

Never permanently delete important business data such as:

- Customers
- Invoices
- Payments
- Inventory History
- Audit Records

Hard Delete requires explicit approval.

---

## 10.9 Transaction Rules

Use database transactions whenever multiple operations must succeed or fail together.

Examples.

- Create Invoice + Invoice Items
- Payment + Ledger Update
- Inventory Movement + Stock Update

Never leave partially completed business operations.

---

## 10.10 Naming Convention

Use consistent naming.

Examples.

Tables:

```text
customers
appointments
inventory_items
payments
```

Columns:

```text
customer_id
created_at
updated_at
deleted_at
```

Avoid inconsistent naming styles.

---

## 10.11 Audit Fields

Every business table should include when applicable.

- id
- created_at
- updated_at
- created_by
- updated_by
- deleted_at
- deleted_by

Auditability is a project requirement.

---

## 10.12 Index Strategy

Indexes should support:

- frequently searched fields
- foreign keys
- unique constraints
- sorting
- filtering

Avoid unnecessary indexes.

Review performance before adding new indexes.

---

## 10.13 Data Modification

Before modifying existing schema:

Review:

- affected modules
- affected APIs
- affected reports
- affected integrations
- existing migrations

Never change schema without impact analysis.

---

## 10.14 Seed Data

Seed data should contain only:

- master data
- lookup data
- development data

Production data must never be included in seed files.

---

## 10.15 Performance Considerations

Database design should minimize:

- duplicate data
- unnecessary joins
- full table scans
- N+1 queries

Optimize based on measurement.

---

## 10.16 Forbidden Actions

Never:

- modify production schema manually
- remove foreign keys without approval
- remove unique constraints without approval
- bypass migrations
- duplicate business data
- use raw SQL unnecessarily

These actions are prohibited.

---

## 10.17 AI Decision Rules

- Which modules depend on these tables?
- Is a migration required?
- Is backward compatibility preserved?
- Is data integrity maintained?

# CHAPTER 11 — API RULES

---

## 11.1 Purpose

This chapter defines the mandatory rules for designing, implementing, documenting, and maintaining APIs within the NIAHAIR ERP project.

The API is the official contract between the backend and all clients.

Every API must remain consistent, predictable, secure, and well documented.

---

## 11.2 API Philosophy

The API represents the business domain.

It is not merely a transport mechanism.

Every endpoint should expose business capabilities rather than database structures.

Design APIs around business operations, not tables.

---

## 11.3 REST Standard

All APIs follow REST principles.

Use resource-based endpoints.

Correct.

```text
GET    /customers

GET    /customers/{id}

POST   /customers

PATCH  /customers/{id}

DELETE /customers/{id}
```

Avoid action-based URLs.

```text
/createCustomer

/updateCustomer

/deleteCustomer
```

---

## 11.4 API Versioning

Every public endpoint belongs to an API version.

Example.

```text
/api/v1/customers

/api/v1/invoices

/api/v1/payments
```

Breaking changes require a new version.

---

## 11.5 Request Validation

Every request must be validated before reaching Business Logic.

Validate:

- required fields
- data types
- enums
- UUIDs
- ranges
- string lengths
- file uploads

Business Rules are not part of request validation.

---

## 11.6 Response Standard

All endpoints must return the same response structure.

Example.

```json
{
  "success": true,
  "message": "Customer created successfully.",
  "data": {}
}
```

Errors should follow the same structure.

---

## 11.7 Error Handling

Never expose internal implementation details.

Return appropriate HTTP status codes.

Examples.

```text
400 Bad Request

401 Unauthorized

403 Forbidden

404 Not Found

409 Conflict

422 Unprocessable Entity

500 Internal Server Error
```

Internal exceptions should be handled by the Global Error Handler.

---

## 11.8 Pagination

List endpoints must support pagination.

Standard query parameters.

```text
?page=1

&pageSize=20
```

Responses should include metadata.

Example.

```json
{
  "data": [],
  "meta": {
    "page": 1,
    "pageSize": 20,
    "total": 250,
    "totalPages": 13
  }
}
```

---

## 11.9 Filtering

Collection endpoints should support filtering where appropriate.

Examples.

```text
?status=ACTIVE

?branchId=...

?employeeId=...
```

Filtering must follow documented Business Rules.

---

## 11.10 Sorting

Support sorting when returning collections.

Example.

```text
?sortBy=name

?order=asc
```

Sorting should be explicit and predictable.

---

## 11.11 Authentication

Protected endpoints require authentication.

Public endpoints must be explicitly documented.

Authentication must be handled by middleware.

---

## 11.12 Authorization

Permissions must be enforced on the backend.

Never rely solely on frontend restrictions.

Every protected endpoint should verify:

- authenticated user
- role
- permission
- ownership (if applicable)

---

## 11.13 DTO Usage

Controllers communicate using DTOs only.

Never expose:

- Prisma models
- internal entities
- database implementation details

DTOs define the public API contract.

---

## 11.14 Idempotency

Safe operations should be idempotent whenever possible.

Examples.

```text
GET

PUT

DELETE
```

Where business operations require retries (such as payments), implement idempotency mechanisms when appropriate.

---

## 11.15 API Documentation

Every endpoint must be documented.

Documentation includes.

- summary
- description
- request
- response
- authentication
- permissions
- examples
- error responses

Swagger/OpenAPI is the official API documentation.

---

## 11.16 API Evolution

When modifying existing endpoints:

- preserve backward compatibility
- document changes
- update tests
- update Swagger

Avoid breaking existing clients.

---

## 11.17 Forbidden Actions

Never:

- expose internal models
- bypass DTOs
- skip validation
- return inconsistent responses
- expose stack traces
- change existing endpoints without documentation

---

## 11.18 AI Decision Rules

Before implementing an API ask:

- Which Business Rule does this endpoint support?
- Is a similar endpoint already available?
- Is the response consistent?
- Is authentication required?
- Is authorization required?
- Does the endpoint require pagination, filtering, or sorting?
- Has Swagger been updated?

Proceed only after answering these questions.

# CHAPTER 13 — STATE MANAGEMENT RULES

---

## 13.1 Purpose

This chapter defines the mandatory rules for managing application state within the NIAHAIR ERP frontend.

The goal is to ensure state remains predictable, minimal, consistent, and easy to maintain.

Always choose the appropriate type of state before implementing new functionality.

---

## 13.2 State Philosophy

State should exist only when necessary.

Do not store data simply because it may be useful later.

Every state should have:

- a clear owner
- a clear purpose
- a clear lifecycle

Minimize state whenever possible.

---

## 13.3 State Categories

Always classify state before implementation.

| State Type | Purpose |
|------------|---------|
| Local State | Temporary UI state |
| Global State | Shared application state |
| Server State | Backend data |
| URL State | Navigation state |
| Persistent State | User preferences and session |

Never mix different categories.

---

## 13.4 State Ownership

Every piece of state must have exactly one owner.

Examples.

```text
Dialog Open

↓

Dialog Component
```

```text
Authenticated User

↓

Auth Store
```

```text
Customer List

↓

TanStack Query
```

Avoid multiple sources of truth.

---

## 13.5 Local State

Use Local State only for temporary UI behavior.

Examples.

- modal visibility
- active tab
- selected row
- expanded panel
- temporary form values

Local State should never be shared across unrelated pages.

---

## 13.6 Global State

Global State should remain small.

Only store data that is required throughout the application.

Examples.

- authenticated user
- active branch
- application theme
- sidebar state
- language

Do not place server data inside Global State.

---

## 13.7 Server State

Backend data belongs to Server State.

Manage it using TanStack Query.

Examples.

- customers
- appointments
- invoices
- inventory
- employees

Do not duplicate Server State inside Zustand.

---

## 13.8 URL State

Navigation-related state belongs in the URL.

Examples.

```text
?page=2

?search=keratin

?status=ACTIVE

?branch=jakarta
```

Users should be able to refresh the page without losing navigation state.

---

## 13.9 Persistent State

Persist only information that improves user experience.

Examples.

- theme
- language
- remembered branch
- user preferences

Never persist sensitive information unless explicitly approved.

---

## 13.10 Derived State

Derived State should be calculated.

Do not store values that can be derived.

Example.

Incorrect.

```text
Invoice Total

↓

Stored separately
```

Correct.

```text
Invoice Total

↓

Calculated from Invoice Items
```

Avoid duplicate calculations stored as state.

---

## 13.11 State Updates

State updates should be predictable.

Always use the official update mechanism.

Never mutate state directly.

State transitions should be easy to trace.

---

## 13.12 State Synchronization

Avoid inconsistent state.

When backend data changes:

- invalidate cache
- refresh queries
- update dependent UI

Do not manually synchronize duplicated state.

---

## 13.13 State Lifetime

Every state has a lifecycle.

Examples.

Local State:

Created

↓

Used

↓

Destroyed

Server State:

Fetched

↓

Cached

↓

Invalidated

↓

Refetched

Understand the lifecycle before implementation.

---

## 13.14 State Performance

Avoid unnecessary re-renders.

Prefer.

- memoization when appropriate
- selective subscriptions
- normalized updates
- efficient query invalidation

Optimize only after measuring.

---

## 13.15 Forbidden Actions

Never:

- duplicate Server State
- store derived values unnecessarily
- mutate state directly
- place Business Logic inside stores
- use Global State as a replacement for Server State

These actions reduce maintainability.

---

## 13.16 AI Decision Rules

Before creating state ask:

- Does this really need state?
- What type of state is it?
- Who owns this state?
- How long should it live?
- Can it be derived?
- Is it already available elsewhere?

Proceed only after answering these questions.

---
